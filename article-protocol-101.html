How Card Transaction Protocols Actually Work: A Walkthrough

Summary

Every card swipe, dip, tap, or online checkout in the world rides on a stack of protocols that has been stable for forty years. The wire format is ISO 8583. The chip layer is EMV. The authorization travels through a card network like Visa or Mastercard to an issuer, which decides yes or no in under a second. Settlement happens later, in batches.

This guide unpacks that stack in plain English. It covers what the protocols are, what the message fields mean, how authorization differs from settlement, and how reversals and chargebacks fit in. Skim the glossary at the bottom when an acronym blurs together.

The Real Protocol Stack

When people say "card transaction," they usually mean the entire chain. But the chain is made of distinct layers, each governed by its own standard. Here's the actual stack from the top of the card down to settlement:

  • EMV — the chip-card standard from EMVCo (Europay, Mastercard, Visa). Defines how the physical chip and the terminal talk to each other.
  • ISO 8583 — the message format used between acquirer, network, and issuer to authorize the transaction.
  • Card network rails — VisaNet, Mastercard MIP, American Express, Discover, JCB, UnionPay, RuPay. The physical and logical infrastructure that routes the ISO 8583 messages.
  • Clearing files — daily batch files (Visa BASE II, Mastercard CIS) that move authorized transactions toward settlement.
  • Settlement — the actual cash movement between banks, typically T+1 or T+2 via central bank rails.
Important. There is no "cloud bypass" layer in this stack. Every legitimate transaction follows this path. Anyone offering to skip steps is either talking about something else or describing fraud.

ISO 8583: The Card Wire Format

ISO 8583 is the international standard that defines what a card transaction message looks like. It was first published in 1987 and has remained the backbone of card payments ever since. Every Visa, Mastercard, Amex, and domestic card switch in the world speaks some flavor of it.

An ISO 8583 message is a binary structure with three parts: a message type indicator (MTI), a bitmap describing which fields are present, and the field values themselves. There can be up to 128 data elements (DEs) in a message, each carrying a specific piece of information like the card number, amount, terminal ID, or response code.

Message Types and Bitmaps

The MTI is a four-digit number that tells the receiver what kind of message this is. The first digit is the version, the second is the class (authorization, financial, reversal, etc.), the third is the function (request, response, advice), and the fourth is the origin.

MTIMeaning
0100Authorization request
0110Authorization response
0200Financial transaction request
0210Financial transaction response
0400Reversal request
0420Reversal advice
0800Network management request
0810Network management response

The bitmap is a 64-bit (or 128-bit if a secondary bitmap is needed) sequence where each bit represents whether a particular data element is present in the message. This is what makes ISO 8583 efficient: only the fields actually used are transmitted.

EMV: The Chip Layer

Below the wire format sits the chip-card protocol itself. EMV defines how the chip on your card communicates with the terminal during a card-present transaction. It generates dynamic cryptograms that change with every transaction, which is what makes chip cards far harder to clone than the old magnetic stripe.

Key EMV Concepts

  • ARQC — Authorization Request Cryptogram. Generated by the chip using a session key, sent to the issuer to prove the card is genuine.
  • ARPC — Authorization Response Cryptogram. The issuer's reply, also cryptographic, which the terminal verifies before completing the transaction.
  • CAPK — Certification Authority Public Keys. The RSA roots stored in the terminal that verify the card's signed data.
  • CVM — Cardholder Verification Method. PIN, signature, no-CVM, or mobile biometric, in priority order defined on the chip.
  • Tokenization — replaces the PAN with a network token (Visa VTS, Mastercard MDES) so the real card number never touches the merchant.
Liability shift. In most regions, if a chip card is used at a chip-capable terminal and fraud occurs, the issuer is liable. If the merchant falls back to magnetic stripe when EMV was available, the merchant is liable. This is why "EMV fallback" transactions are flagged aggressively by fraud systems.

The Full Transaction Flow

Here's what actually happens when you tap your card at a coffee shop, in plain sequence:

  1. Cardholder presents card — chip read, contactless tap, or card-not-present checkout submits PAN and expiry.
  2. Terminal/gateway constructs ISO 8583 0100 — authorization request with PAN, amount, terminal ID, EMV cryptogram (if chip).
  3. Acquirer processor receives the request — runs basic validation, fraud scoring, then forwards to the card network.
  4. Network routes to the issuer — based on the BIN (first 6 digits of the PAN).
  5. Issuer decides — checks available balance, daily limits, fraud rules, sanctions lists, EMV cryptogram validity. Returns approve or decline.
  6. 0110 response travels back — issuer → network → acquirer → terminal. End-to-end usually under one second.
  7. Terminal prints receipt — transaction is authorized but not yet cleared or settled.
  8. End of day batch — acquirer submits clearing files to the network via Visa BASE II or Mastercard CIS.
  9. Network presents to issuer — issuer confirms the clearing record matches the earlier authorization.
  10. Settlement — funds move between issuer and acquirer banks via central bank settlement rails. Usually T+1 or T+2.
  11. Merchant gets paid — the acquirer credits the merchant's account, minus interchange and assessment fees.

Authorization vs Clearing vs Settlement

Three distinct events. Beginners often confuse them.

StepWhat HappensWhen
Authorization Issuer says "yes, this transaction is approved" and places a hold on the cardholder's available balance. At the moment of the transaction (real-time, sub-second)
Clearing The merchant submits the transaction to the network for processing. Authorization holds become real charges. End of day, batch
Settlement Money actually moves between issuer and acquirer banks. T+1 or T+2 typically
Why it matters. A merchant getting a 0110 "approved" response doesn't have the money yet. They have a promise. Real cash arrives at settlement, days later. This is where chargebacks become possible — they're disputes against transactions that have authorized but not yet been finalized in the customer's favor.

Reversals and Chargebacks

Things go wrong. The protocol stack has defined paths for each kind of correction.

Reversal (0400 / 0420)

A reversal happens when the merchant or acquirer cancels a transaction before it clears. Common reasons: terminal timeout, customer cancels, duplicate charge detected. The 0400 message asks the issuer to release the authorization hold.

Refund

A new transaction in the opposite direction. Initiated by the merchant after clearing, usually for returned goods. The cardholder sees a credit on their statement.

Chargeback

A dispute initiated by the cardholder through their issuer. The issuer pulls funds back from the acquirer until the merchant either provides evidence (representment) or accepts the loss. Each card network has detailed chargeback reason codes governing what's disputable and what's not.

Frequently Asked Questions

Why is there both an authorization and a settlement step?

Speed and risk. Authorization happens in under a second so the merchant knows immediately whether to release goods. Settlement is a slower, batch-oriented process because moving real money between banks takes coordination and reconciliation. Splitting the two lets the card networks be both fast at the point of sale and accurate at the point of payment.

Can a transaction be authorized but never settle?

Yes. If the merchant never submits the transaction for clearing (within a defined window, usually 7 to 30 days), the authorization hold expires and the funds return to the cardholder's available balance. This is rare but not impossible.

What's the difference between a credit card and a debit card transaction at the protocol level?

At the ISO 8583 layer, very little. The processing code (DE3) and the BIN both tell the network whether the card is credit, debit, or prepaid. The issuer's response logic differs (debit checks the actual balance, credit checks the available credit line), but the wire format is the same.

What protocol does an online checkout use?

The same ISO 8583 stack on the back end. The front end uses a payment gateway API (Stripe, Adyen, Razorpay) over HTTPS, but once the gateway receives the card details, it constructs an ISO 8583 message to send through the network. The customer never sees the underlying protocol, but it's the same rails.
